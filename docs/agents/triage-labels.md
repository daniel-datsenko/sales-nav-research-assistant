# Agent Triage Labels

## Canonical roles

Each triaged issue should carry exactly one category label and one state label.

## Category labels

- `bug` — broken behavior, regression, failing safety invariant, or performance issue.
- `enhancement` — new feature, workflow improvement, documentation improvement, refactor, or operating-system improvement.

## State labels

- `needs-triage` — maintainer or Supervisor needs to evaluate scope and state.
- `needs-info` — waiting on reporter or Operator for specific missing information.
- `ready-for-agent` — fully specified AFK Slice; an implementation agent can pick it up without additional human decisions.
- `ready-for-human` — HITL Slice; needs Operator judgment, external access, live Sales Navigator validation, or product/risk approval.
- `wontfix` — will not be actioned.

## State transition rules

- Unlabeled issues normally enter `needs-triage`.
- `needs-triage` can move to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`.
- `needs-info` returns to `needs-triage` once the reporter answers.
- `ready-for-agent` requires an Agent Brief or equivalent acceptance criteria.
- `ready-for-human` should explain why AFK work is unsafe or insufficient.
- If an issue has conflicting state labels, stop and ask before changing it.

## Ready-for-agent bar

An issue can be `ready-for-agent` only when it has:

- clear objective and non-goals
- exact safety constraints
- relevant files/modules or discovery instructions
- acceptance criteria
- deterministic verification commands
- no required live Sales Navigator mutation
- no unresolved product/risk decision
- no dependency on private runtime artifacts unless sanitized samples are provided

## AI-generated comments

When an agent posts triage notes or an Agent Brief to GitHub, start the comment with:

```markdown
> *This was generated by AI during triage.*
```
