# ADR 0003: Sanitize execution-gate commands before display

## Status

Accepted

## Context

Autoresearch artifacts can contain execution gates, research loop plans, and command templates. Those artifacts may be stale, malformed, user-supplied, or generated by prior code versions. Operator-facing reports turn this evidence into copyable next actions, so unsafe command leakage can become an accidental live mutation path.

The risk is highest when a non-live decision still contains `--live-save`, `--live-connect`, background-connect allowances, or command names that imply mutation without obvious flags.

## Decision

All commands rendered from Execution Gates, Research Loop Plans, Gate Reports, and Supervisor Runbooks must be sanitized before display.

- Read-only report commands must refuse live flags at CLI entry.
- Non-live decisions must never render copyable commands containing `--live-save`, `--live-connect`, `--allow-background-connects`, or implicit live-mutation commands.
- Unsafe commands in non-live contexts should be replaced with an explicit marker such as `unsafe_command_suppressed` plus a reason.
- Eligible live-save decisions may render a supervised live command only when the report also states that human approval is required and no automatic execution will occur.
- Report renderers must not blindly trust artifact command strings.

## Consequences

- Gate/report tests should include malformed artifacts with unsafe commands and prove the unsafe text is not leaked for non-live decisions.
- New report surfaces should reuse the same sanitization semantics instead of reimplementing ad hoc filtering.
- Agents should treat rendered commands as recommendations unless the command is dry-safe and the current task explicitly authorizes execution.
